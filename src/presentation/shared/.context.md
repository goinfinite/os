Cross-presentation-layer helpers consumed by both the API and UI liaisons. Centralizes utilities that must produce identical behaviour regardless of whether the caller is an HTTP route or a UI state handler.

## Summary

- helper/ — Service availability inspector and other cross-presentation utilities

## Constraints

- MUST NOT depend on API or UI specific packages
- MUST remain side-effect free at module-init time; all state must come from explicit constructor parameters
- MUST use shared DTOs and value objects for input/output to keep the API and UI surfaces consistent
- MUST NOT introduce new database services beyond what each helper explicitly requires

## Guidance

- SHOULD prefer extending an existing helper over creating a parallel one with overlapping responsibility
- SHOULD return plain value objects or boolean flags (e.g., `IsAvailable`) so callers can compose their own liaison responses
- SHOULD keep constructors explicit — pass dependencies as parameters rather than reaching into globals
